Phase 4 – Performance, Reliability & Production Readiness

You are continuing work on an existing Python Code Visualizer project.

Read First

Do not assume anything.

First inspect the entire repository before making changes.

Understand the architecture.

Identify how the frontend and backend currently communicate.

Understand the tracer, heap serialization, snapshot generation, playback system, renderers, timeline, variable panels, call stack, and API.

Only after understanding the project should you make a plan.

Then execute the plan incrementally.

Phase 4 Goals
1. Performance

Inspect the execution engine for unnecessary work.

Examples include

repeated deep copies
repeated heap traversal
unnecessary serialization
duplicated work every snapshot
expensive loops

Improve performance without changing external behavior.

2. Memory Usage

Reduce memory consumption.

Avoid storing unnecessary duplicate data.

If snapshots can safely share immutable data, use structural sharing where appropriate.

Do not break snapshot immutability.

3. Code Cleanup

Remove dead code.

Remove duplicate implementations.

Consolidate duplicate helpers.

Remove unused imports.

Remove unused modules only if they are truly unused.

Document every removal.

4. Backend Reliability

Improve robustness.

Handle edge cases cleanly.

Improve timeout handling.

Improve recursion handling.

Improve execution limit reporting.

Improve malformed request handling.

Improve API validation.

5. Frontend Reliability

Improve error handling.

Differentiate

backend unavailable
Python runtime error
timeout
recursion limit
step limit
syntax error

Each should have its own user-friendly UI.

6. Testing

Expand automated tests.

Cover

performance
API contract
edge cases
malformed requests
execution limits
visualization compatibility
regression tests

Existing tests must continue passing.

7. Documentation

Update

README

architecture documentation

Include

system architecture

API contract

development workflow

Constraints

Do NOT redesign the architecture.

Do NOT rewrite the tracer.

Do NOT change the API contract unless absolutely necessary.

Keep frontend compatible.

Keep all existing functionality working.
